# AI INTEGRATION

## 1. AI Integration Requirements

Integrating AI features into data analysis: this can be a standalone application, integrated into a dashboard, or by creating an API for other applications to use.

Creating an API for other applications to use is encouraged.  

### 1.1 Basic Principles

AI: Plays the role of  assisting in proposing ideas.  writing code based on user requirements.  presenting analysis results with data, images, and charts provided by humans; it is not allowed to arbitrarily add other data or images.  

Human: Plays the role of  guiding, proposing analysis requirements, and making decisions to execute code.  if the human does not know or has no ideas on how to analyze, they can ask the AI to provide suggestions and methods to choose from.  

Code execution: must be on the user's local environment. Execution on an online environment is not allowed.  

## 2. The "No Background Execution" Principle

AI is not allowed to arbitrarily change original data or silently run algorithms.  

Mandatory code display: Whenever the AI generates code requested by a human, it must be clearly displayed.  

Natural language explanation: Right above the code, the AI must attach comment lines explaining it in natural language: "This code will delete 15 rows with NULL values in the Revenue column, using the Pandas dropna() function."  

### 2.1 Approval Principle

Demonstrates the "Decision Maker" role.  

Pending state: Code initially generated by AI will be in a "Pending approval" state. Humans can change, add to, or remove this code.  

Editing: Humans have the right to directly intervene and type to edit the parameters in the code the AI just wrote (e.g., changing the outlier limit from 3 standard deviations down to 2).  

Acceptance & execution: The code is only executed and returns results/charts if and only if the human approves.  

### 2.2 Storage Principle

The system must store all requests, source code, analysis results, and explanations so they can be retrieved later.  

## 3. In Q&A

Groups will be required to use their developed AI module to answer analysis questions based on their group's data and pre-prepared analysis questions, in order to avoid going off-topic. However, they may still be asked off-topic questions to evaluate the AI's flexibility. Note that the minimum number of questions will equal the number of group members.  

## 4. In the Report

Must summarize the process of using the AI, the requirements set, the results received, the changes made, and comments regarding the AI.  

## 5. Design Suggestions

Divide the architecture into API and Frontend.  

### 5.1 User Interface & Interaction (Frontend)

Frontend development technology is optional (can use Streamlit, Gradio, React, or interactive Dashboard tools, etc.). Groups only need to ensure basic functionalities. Groups are encouraged to create beautiful, intuitive, and user-friendly interfaces.  

Request receiving function: For example, a chat box or input form for users to send analysis/coding requests to the AI.  

Source code viewing & editing function.  

Approval function.  

Result display function.  

### 5.2 API Section  

AI API (mandatory):  

Receive requests from the Front-end.  

Send along the context (e.g., existing data structure) to the AI model (Gemini/OpenAI) (local models are permitted).  

Require the AI to return both the code and the corresponding explanation; or the presented results.  

Execution API (mandatory):  

Receive the code snippet that has been "edited" and "approved" by the human from the frontend.  

Execute the code directly on the local machine's data.  

Collect results (chart images, data tables, logs) to return to the frontend for display.  

Logs API (mandatory):  

Store all requests, source code, analysis results, and explanations.
